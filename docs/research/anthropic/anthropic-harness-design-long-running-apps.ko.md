## 한줄 요약

이 글은 좋은 결과를 만드는 핵심이 모델 자체만이 아니라, 모델이 일하는 방식 전체를 감싸는 `하네스(harness)` 설계에 있다는 점을 보여준다. Anthropic은 프런트엔드 디자인 생성과 장시간 자율 앱 개발이라는 두 문제를 다루면서, 단일 에이전트보다 `계획자(planner)`, `생성자(generator)`, `평가자(evaluator)`를 분리한 구조가 더 높은 품질의 결과를 만든다고 주장한다.

## 글의 문제의식

저자는 두 가지 과제를 동시에 다뤘다.

1. Claude가 더 좋은 프런트엔드 디자인을 만들게 하는 것
2. 사람 개입 없이 완성도 높은 애플리케이션을 장시간 자율적으로 개발하게 하는 것

기존에도 프롬프트 엔지니어링과 하네스 설계로 성능 향상은 있었지만, 어느 지점 이후에는 개선이 잘 일어나지 않았다. 특히 복잡한 작업에서는 다음 문제가 반복되었다.

- 긴 작업을 수행할수록 모델이 맥락을 잃고 작업 품질이 흔들린다.
- 모델은 자신이 만든 결과물을 스스로 평가할 때 지나치게 관대해진다.
- 디자인처럼 정답이 명확하지 않은 문제에서는 "괜찮아 보인다" 수준에서 멈추기 쉽다.
- 코딩에서도 겉보기에는 그럴듯하지만 실제 사용 시 깨지는 기능이 자주 남는다.

이 글의 핵심은 이런 한계를 모델 파라미터가 아니라 작업 구조 설계로 보완하는 데 있다.

## 핵심 개념 1: 왜 순진한 단일 에이전트 방식이 무너지는가

### 1. 긴 작업에서의 맥락 붕괴

저자는 긴 작업에서 모델이 컨텍스트 윈도우가 차오를수록 일관성을 잃는다고 본다. 특히 일부 모델은 컨텍스트 한계를 의식한 듯 작업을 너무 일찍 마무리하려는 경향을 보였는데, 글에서는 이를 `context anxiety`라고 설명한다.

여기서 중요한 구분은 다음과 같다.

- `compaction`: 기존 대화를 요약해서 같은 에이전트가 계속 일하게 함
- `context reset`: 새 에이전트를 띄우고, 이전 상태를 구조화된 산출물로 넘겨받게 함

Anthropic의 이전 실험에서는 Sonnet 4.5가 긴 작업에서 이런 문제를 강하게 보여, 단순 요약만으로는 부족했고 컨텍스트 리셋이 필요했다고 한다.

### 2. 자기평가의 구조적 한계

모델은 자신이 만든 결과를 평가할 때 비판적이지 못하다. 특히 디자인처럼 주관적 판단이 필요한 영역에서는 더 심하다. 따라서 "만드는 역할"과 "판단하는 역할"을 분리해야 한다는 것이 저자의 주장이다.

핵심은 평가자를 따로 둔다고 해서 자동으로 엄격해지지는 않지만, 적어도 별도 평가자 프롬프트를 튜닝하는 편이 생성자가 자기 자신을 냉정하게 비판하게 만드는 것보다 훨씬 쉽다는 점이다.

## 핵심 개념 2: 프런트엔드 디자인을 어떻게 '채점 가능한 문제'로 바꿨는가

프런트엔드 디자인 실험은 이 글의 출발점이다. 저자는 기본 상태의 Claude가 기능적으로는 멀쩡하지만 시각적으로 평범하고 안전한 결과물로 수렴한다고 본다.

이를 해결하기 위해 디자인 품질을 네 가지 기준으로 나눴다.

1. 디자인 완성도: 색, 타이포, 레이아웃, 이미지가 하나의 정체성으로 묶이는가
2. 독창성: 템플릿이나 라이브러리 기본값을 복제한 느낌이 아닌가
3. 공예적 완성도: 간격, 계층, 대비, 조화 같은 기본기가 탄탄한가
4. 기능성: 사용자가 헤매지 않고 목표 행동을 수행할 수 있는가

여기서 저자는 `공예적 완성도`와 `기능성`보다 `디자인 완성도`와 `독창성`에 더 높은 비중을 두었다. 이유는 Claude가 기본적으로 기능성과 기본기 쪽은 어느 정도 만족하지만, 디자인적으로는 쉽게 무난한 결과로 후퇴하기 때문이라고 본다.

즉, 이 실험의 포인트는 "아름다움"을 직접 채점하는 것이 아니라, 좋은 디자인이 갖춰야 할 원칙을 평가 가능 항목으로 바꾸는 데 있다.

## 프런트엔드용 생성자-평가자 루프

구조는 비교적 단순하다.

- 생성자: 사용자 프롬프트를 바탕으로 HTML/CSS/JS 기반 UI 생성
- 평가자: Playwright MCP로 실제 페이지를 탐색하고 점수와 상세 피드백 작성
- 반복: 평가 결과를 다시 생성자에게 넘겨 다음 버전 생성

저자는 한 번 생성하고 끝내지 않고, 보통 5회에서 15회까지 반복했다고 설명한다. 각 반복은 실제 페이지를 띄우고 탐색하는 과정이 포함되어 시간이 오래 걸렸고, 전체 실행은 최대 4시간까지 갔다.

중요한 점은 생성자가 매 반복마다 단순 수정만 하는 것이 아니라, 현재 방향을 더 다듬을지 아니면 완전히 다른 미학적 방향으로 피벗할지도 결정하도록 했다는 점이다. 그래서 어떤 경우에는 점진적으로 좋아졌고, 어떤 경우에는 중간에 미학적 방향이 크게 바뀌었다.

저자가 제시한 흥미로운 관찰은 다음과 같다.

- 평가 기준 문구 자체가 산출물의 스타일을 강하게 유도했다.
- 반복할수록 평균 점수는 좋아졌지만, 항상 마지막 결과가 최고는 아니었다.
- 반복이 진행될수록 구현 복잡도와 야심도도 커졌다.
- 평가 루프가 돌기 전 첫 결과물조차, 기준 없는 기본 생성보다 더 나았다.

즉, 평가자는 사후 검사기일 뿐 아니라, 생성자의 탐색 공간 자체를 재형성하는 장치로 작동했다.

## 장시간 풀스택 코딩으로 확장한 방식

프런트엔드 실험에서 얻은 패턴을 저자는 풀스택 애플리케이션 생성으로 확장한다. 여기서는 구조가 `3에이전트 체계`로 바뀐다.

### 1. Planner

짧은 사용자 요청을 더 큰 제품 명세로 확장한다. 저자는 처음부터 구현 세부사항까지 빽빽하게 적게 하기보다, 제품 맥락과 상위 수준 기술 설계 중심으로 명세를 확장하도록 했다. 너무 이른 단계에서 세부 기술 결정을 고정하면, 잘못된 전제가 뒤 단계 전체에 퍼질 수 있기 때문이다.

### 2. Generator

실제 앱을 구현하는 역할이다. 초기 버전에서는 스프린트 단위로 기능을 하나씩 구현했고, 기술 스택은 React, Vite, FastAPI, SQLite에서 이후 PostgreSQL로 확장되었다. 각 스프린트가 끝나면 자체 점검 후 QA로 넘겼다.

### 3. Evaluator

Playwright MCP를 써서 실제 사용자가 하듯 앱을 클릭하고, UI 동작, API 엔드포인트, 데이터베이스 상태를 점검했다. 평가 기준은 제품 깊이, 기능성, 시각 디자인, 코드 품질 등으로 구성했고, 어느 한 기준이라도 임계값 아래면 그 스프린트는 실패 처리했다.

## 스프린트 계약(sprint contract)의 의미

이 글에서 특히 중요한 설계는 스프린트 시작 전 생성자와 평가자가 `완료 기준`을 합의하는 단계다. 제품 명세는 일부러 고수준으로 유지되므로, 실제 구현 가능한 작업 단위와 테스트 가능한 검증 조건 사이를 이어주는 다리가 필요했다.

이 계약은 다음 역할을 한다.

- 구현 전에 "이번 라운드에서 무엇을 끝냈다고 볼지"를 명확히 함
- 생성자가 자기 마음대로 범위를 축소하는 일을 막음
- 평가자가 뒤늦게 다른 기준을 들이밀지 않게 함
- 모호한 제품 요구를 테스트 가능한 조건으로 변환함

에이전트 간 소통도 대화가 아니라 파일 기반으로 처리했다. 한 에이전트가 파일을 쓰면 다른 에이전트가 읽고 응답 파일을 생성하는 방식이다. 이는 긴 세션에서 맥락을 안정적으로 넘기는 데 유리한 선택으로 보인다.

## 첫 번째 실험: 레트로 게임 메이커

비교 실험은 동일한 짧은 프롬프트에서 시작했다.

- 단일 에이전트: 약 20분, 약 9달러
- 전체 하네스: 약 6시간, 약 200달러

비용은 20배 이상 들었지만, 결과 품질 차이는 컸다고 한다.

### 단일 에이전트 결과

- 겉보기에는 그럴듯했지만 핵심 기능이 실제로 잘 작동하지 않음
- 레이아웃이 비효율적이고 화면 사용이 서툼
- 사용자 워크플로 안내가 부족함
- 게임 플레이 핵심 로직이 깨져 있었음

### 전체 하네스 결과

- 1줄 프롬프트가 16개 기능, 10개 스프린트 수준의 제품 명세로 확장됨
- 스프라이트 애니메이션, 행동 템플릿, 사운드, AI 기반 스프라이트/레벨 생성, 공유 가능한 export 같은 범위까지 포함
- UI 완성도와 작업 흐름이 더 좋아졌고, 실제 플레이도 가능했음
- 여전히 엣지 케이스와 상식 수준 UX 문제는 남아 있었음

핵심은 완벽함이 아니라 `핵심 사용자 가치가 실제로 작동하는 수준까지 끌어올렸다`는 데 있다.

## QA 평가자는 어떻게 강해졌는가

저자는 기본 상태의 Claude가 QA 역할에 약하다고 솔직히 적는다. 실제 버그를 발견하고도 "그 정도는 괜찮다"고 넘기거나, 피상적으로만 테스트하는 경향이 있었다는 것이다.

그래서 저자는 평가 로그를 읽고, 자신의 판단과 어긋나는 사례를 모아 평가자 프롬프트를 계속 수정했다. 즉, 평가자도 한 번에 완성된 것이 아니라 반복적으로 훈련된 운영 자산이었다.

원문에서 소개한 실제 실패 사례는 다음 종류였다.

- 드래그해서 사각형 영역을 채워야 하는 툴이 시작점과 끝점만 찍는 버그
- 엔티티 삭제 조건식이 잘못되어 선택은 되지만 삭제가 안 되는 버그
- FastAPI 라우팅 순서 문제 때문에 `reorder`가 정수 ID로 해석되는 버그

이 사례들이 보여주는 것은 평가자가 단순한 "느낌 평가"를 넘어서, 실제 구현 오류를 구체적으로 지적하는 디버깅 파트너 역할을 했다는 점이다.

## 두 번째 단계: 하네스를 단순화하는 실험

첫 버전은 효과적이었지만 무겁고 느리고 비쌌다. 저자는 여기서 중요한 원칙을 제시한다.

하네스의 각 구성요소는 "모델이 혼자서는 못할 것"이라는 가정을 담고 있으므로, 그 가정은 지속적으로 재검증해야 한다.

특히 Opus 4.6이 등장하면서 상황이 달라졌다. 모델이 더 오래 일하고, 더 잘 계획하고, 더 큰 코드베이스에서 더 안정적으로 움직일 수 있게 되자, 예전 하네스 구성 중 일부는 더 이상 필수가 아닐 수 있었다.

### 제거한 것: 스프린트 구조

업데이트된 하네스에서는 스프린트 구조를 제거했다. Opus 4.6은 예전보다 더 긴 작업을 스스로 버틸 수 있었기 때문이다.

하지만 모든 요소를 제거하지는 않았다.

- Planner는 여전히 중요했다.
  생성자가 원시 프롬프트만 받으면 범위를 작게 잡는 경향이 있었기 때문이다.
- Evaluator는 조건부로 여전히 가치가 있었다.
  모델 단독으로도 해결 가능한 범위에서는 오버헤드가 되지만, 능력 경계선 근처의 어려운 작업에서는 여전히 큰 도움을 줬다.

즉, 평가자는 항상 켜야 하는 고정 부품이 아니라, 현재 모델의 기본 성능과 작업 난이도 사이 거리로 판단해야 하는 선택적 장치라는 것이 저자의 결론이다.

## 업데이트된 하네스 실험: 브라우저 기반 DAW

새 하네스는 브라우저 기반 디지털 오디오 워크스테이션(DAW) 생성으로 시험되었다.

- 총 실행 시간: 약 3시간 50분
- 총 비용: 약 124.70달러
- 그중 첫 번째 빌드 라운드만 2시간 7분 동안 일관되게 진행됨

이 결과는 "이전에는 스프린트 분해가 필수였지만, 새 모델에서는 더 긴 연속 작업이 가능해졌다"는 점을 보여준다.

QA는 여기서도 여전히 의미가 있었다. 첫 평가에서는 다음과 같은 본질적 문제를 잡아냈다.

- 타임라인에서 클립을 드래그/이동할 수 없음
- 신스 노브나 드럼 패드 같은 악기 UI가 없음
- EQ 곡선 같은 시각적 이펙트 편집기가 없음

두 번째 평가에서도 다음과 같은 미완성이 지적되었다.

- 오디오 녹음이 실제 마이크 캡처 없이 토글만 됨
- 클립 리사이즈 및 분할 미구현
- 이펙트 표현이 숫자 슬라이더 수준에 머묾

결과적으로 최종 앱은 전문 음악 툴 수준은 아니었지만, 브라우저 안에서 동작하는 어레인지 뷰, 믹서, 트랜스포트, 그리고 에이전트 기반 작곡 흐름까지 갖춘 기능적 골격을 만들었다고 평가한다.

## 이 글이 실무에 주는 시사점

### 1. 좋은 에이전트 시스템은 좋은 프롬프트보다 좋은 역할 분리가 중요하다

이 글에서 가장 설득력 있는 부분은 생성과 평가를 분리했을 때 품질이 실제로 올라갔다는 점이다. 특히 디자인처럼 정답이 애매한 문제에서 이 분리는 매우 강력하다.

### 2. 평가 기준을 먼저 설계해야 한다

모델 품질을 올리려면 "좋은 결과가 무엇인가"를 사람이 먼저 구조화해야 한다. 디자인에서든 QA에서든, 평가 기준이 없는 반복은 막연한 재생성에 그치기 쉽다.

### 3. 긴 작업일수록 상태 전달 방식이 중요하다

컨텍스트 리셋, 파일 기반 핸드오프, 명시적 계약 문서 같은 구조화된 산출물은 장시간 작업에서 일관성을 유지하는 핵심 장치다.

### 4. 하네스는 모델 발전에 따라 계속 바뀌어야 한다

어제 필요한 보조 장치가 오늘은 불필요한 오버헤드일 수 있다. 따라서 하네스는 한 번 설계하고 끝나는 시스템이 아니라, 새 모델이 나올 때마다 `무엇이 아직도 성능을 떠받치고 있는지` 재평가해야 하는 운영 체계다.

### 5. 비용 대비 효과 판단이 필요하다

이 글의 방식은 분명히 강력하지만, 시간과 토큰 비용이 매우 크다. 따라서 모든 작업에 적용하기보다 아래 상황에 우선 쓰는 것이 현실적이다.

- 실패 비용이 큰 장기 작업
- 기능 완성도가 중요한 자율 코딩
- 디자인/제품 품질 같은 주관적 요소가 중요한 작업
- 단일 패스 생성으로는 안정적 결과가 잘 나오지 않는 문제

## 비판적으로 볼 지점

이 글은 매우 흥미롭지만, 몇 가지는 함께 고려할 필요가 있다.

### 1. 일반화 가능성

사례가 인상적이지만, 여전히 몇 개의 데모 중심이다. 다양한 도메인과 팀 환경에서도 같은 비용 대비 효과가 재현되는지는 별도 검증이 필요하다.

### 2. 평가자의 한계

평가자를 분리해도 결국 평가자도 LLM이다. 저자 스스로 인정하듯 QA는 여전히 피상적일 수 있고, 특히 음악처럼 모델이 직접 감각적으로 검증할 수 없는 영역에서는 한계가 뚜렷하다.

### 3. 운영 비용

4시간, 6시간, 100달러 이상의 실험은 연구 환경에서는 가능해도 일반적인 실무 팀에서는 자주 돌리기 어렵다. 이 구조를 실전 배포하려면 실패 시 조기 종료, 단계별 비용 상한, 휴먼 체크포인트 같은 운영 장치가 추가로 필요해 보인다.

## 결론

이 글은 "더 좋은 모델이 나오면 하네스는 덜 중요해진다"는 단순한 이야기를 하지 않는다. 오히려 모델이 좋아질수록 하네스의 형태가 바뀐다고 말한다.

예전에는 컨텍스트 리셋과 스프린트 분해가 필수였다면, 더 강한 모델에서는 그 일부를 제거할 수 있다. 대신 더 높은 수준의 계획, 평가, 역할 분리, 그리고 모델 능력 경계선 근처 문제를 공략하는 새 조합이 중요해진다.

정리하면 이 글의 메시지는 다음과 같다.

- 모델 성능 향상만 기다리지 말고 작업 구조를 설계하라.
- 생성자와 평가자를 분리하라.
- 주관적 품질도 평가 가능 기준으로 바꿔라.
- 하네스는 고정 구조가 아니라 모델 세대에 따라 재설계해야 하는 시스템이다.

이 관점은 앞으로의 에이전트 코딩, 자동 QA, 디자인 생성, 장시간 자율 실행 시스템 전반에 꽤 직접적인 영향을 주는 실무적 통찰이라고 볼 수 있다.
